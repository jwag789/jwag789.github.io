In this blog post I will explain what Ruby on Rails is.

Rails

Ruby on Rails is something I constantly hear about when looking up web development. It is a web application framework, which runs through Ruby and is open source. Rails is a full-stack framework, which gathers libraries useful for the developer, talks to the database, and renders templates. Rails uses the model-view-controller (MVC) pattern in order to organize application programming. A model will typically map to a table in a database and a Ruby file. A controller responds to external requests from the web server to the application. A view in Ruby is an erb file, compiled to HTML at run-time. Rails also includes things to make tasks easier such as scaffolding, and can be integrated with other gems to provide a great development environment. Ruby also uses JavaScript libraries Prototype and Script.aculo.us for Ajax. Rails is seperated into different packages, including ActiveRecord which is a system for database access, ActiveResources which provides web services, ActionPack, ActiveSupport, and ActionMailer. Being open source, many people make plugins as well. Rails can be installed using RubyGems, a package manager, and deployed with a database server and a web server.

History / Philosophies

In July 2004, David Heinemeier Hansson released Rails as an open source, and shared commit rights 7 months later. In August 2006, Apple announced that it would ship Ruby on Rails with their OS, Leopard. This was a huge landmark for Rails. In March 2009, Rails version 2.3 was released with major developments, affecting templates and reusing application pieces. The latest version is 4.1, released just in April 8 of this year. Many things have been added such as Turbolinks, Live Streaming, and things like jQuery as default JavaScript library, and making the development mode faster.

Throughout time however, Rails has maintained certain design principles that are emphasized. Convention over Configuration is one, where when naming classes and their corresponding tables, you should typically use the same name even if it doesn't sound right. Also DRY (Don't repeat yourself) is a big principle that is followed, making the code as clean as possible. Another thing that should be followed is putting most of the logic into the model rather than the controller.
